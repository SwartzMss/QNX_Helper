# 交叉编译与调试

## 关注重点

QNX 学习过程中，编译和调试往往比“写代码”本身更容易卡住，因此建议专门维护这一页。

很多时候代码本身并不复杂，真正花时间的是下面这些问题：

- 为什么编过了但程序起不来
- 为什么部署后缺依赖
- 为什么进程在但功能不工作
- 为什么线程卡住却没有明显报错

把这类内容单独整理成一页，长期价值很高。

## 可记录的内容

- `qcc` 的基础用法
- 不同架构目标的编译参数
- 可执行文件部署方式
- 远程调试方法
- 日志查看和故障定位流程

## 示例记录结构

### 编译

- 工具链选择
- 编译选项说明
- 常见链接错误

### 部署

- 如何把程序放到目标机
- 如何设置可执行权限
- 如何确认依赖库完整

### 调试

- 如何查看进程状态
- 如何定位卡死线程
- 如何收集现场信息

## 推荐排查顺序

如果程序运行异常，建议尽量按固定顺序排查：

1. 先确认程序是否真的启动成功。
2. 再确认依赖文件、路径和权限是否正确。
3. 然后观察进程和线程状态。
4. 如果涉及通信，再检查网络、服务端口或消息链路。
5. 最后再回到代码逻辑本身。

这样做的好处是先排掉环境和部署问题，避免一上来就怀疑代码。

## 调试时建议保留哪些信息

- 目标环境版本
- 可执行文件版本
- 启动命令
- 错误现象
- `pidin` 观察结果
- 相关日志或终端输出

## 常见问题类型

### 能编译，不能运行

优先检查：

- 目标架构是否匹配
- 动态库是否齐全
- 文件权限是否正确
- 程序是否部署到了预期路径

### 进程存在，但功能异常

优先检查：

- 线程是否阻塞
- 是否在等待消息或 IO
- 关键服务是否真的可用
- 优先级是否影响调度

### 网络或服务通信异常

优先检查：

- 接口地址是否正确
- 端口是否监听
- 服务端是否已启动
- 防火墙或链路是否影响连接

## 后续建议

把每次真实遇到的构建错误整理成案例，而不是只记最终命令。
