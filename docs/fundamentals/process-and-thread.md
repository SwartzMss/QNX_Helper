# 进程与线程

## 为什么这部分重要

QNX 的很多调试工作都会直接落到进程、线程、优先级和调度行为上，因此这里是后续学习的基础。

很多“程序没响应”“服务看起来还在但功能不工作”“CPU 占用异常”“调用卡住”这类问题，最后都要回到线程状态和调度行为上看。

## 建议先掌握的点

- 进程和线程的区别
- 主线程与工作线程的职责划分
- 优先级设置的影响
- 调度异常时如何观察系统状态

## 可以先用什么方式理解

### 进程

进程更像资源和职责的边界。一个服务、一个应用或一个系统组件，通常会以进程的形式存在。看进程时，你更关心：

- 它是谁
- 它负责什么
- 它有没有启动成功
- 它占用了哪些资源

### 线程

线程更像真正执行工作的单位。看线程时，你更关心：

- 它现在在做什么
- 它是不是被阻塞了
- 它的优先级是否合理
- 它是不是在等待某个事件或消息

## 为什么线程比想象中更重要

在嵌入式和实时系统里，“程序存在”不代表“功能正常”。一个进程即使还活着，内部关键线程也可能已经：

- 阻塞在消息等待上
- 卡在锁竞争上
- 因优先级设置不合理而得不到调度
- 死循环导致占满 CPU

所以排查问题时，只看“进程在不在”通常不够。

## 常见学习切入点

### 进程

- 地址空间
- 资源隔离
- 服务组织方式

### 线程

- 并发执行单元
- 优先级与调度
- 阻塞与唤醒

## 调试时建议优先观察什么

1. 进程是否存在，是否符合预期启动顺序。
2. 关键线程是否还在运行，还是已经阻塞。
3. 阻塞点是在消息、锁、IO 还是 sleep。
4. 优先级是否与任务实时性要求匹配。

## 学习建议

- 把“线程状态”当成日常排查入口
- 遇到卡顿问题时，不要只看日志，也要看线程
- 尽量把命令观察结果和代码逻辑联系起来理解

## 可补充的命令示例

```bash
pidin threads
```

```bash
pidin -p <pid>
```
